## 事实 1: WEIGHTS 合计 80 是有意设计

[agents/recommendation_agent.py](../agents/recommendation_agent.py) line 67:

```python
WEIGHT_TOTAL = sum(WEIGHTS.values())  # 80
```

作者明确知道，配合归一化 `weighted_sum * 100 / WEIGHT_TOTAL` 使用。**不是 bug**。

任何反推权重提议（如新方案 100 合计）必须同步改归一化分母，否则 score 会无声地缩放。

## 事实 2: dimensions_detail JSON 是 nested 结构

实测 `recommendations.dimensions_detail` JSON 顶层 keys = `{dimensions, phase1, phase3, counter_card}`，**不是 flat 结构**。

正确结构:

```json
{
  "dimensions": {
    "d1": {"score": 100, "weight": 15, "weighted": 15.0, ...},
    "d2": {...},
    ...
    "d6": {...}
  },
  "phase1": {...},
  "phase3": {"delta": 5, "notes": [...]},
  "counter_card": {...}
}
```

**未来重算 / 复盘脚本必须按 nested 解析**（`data['dimensions']['d1']['score']`），不能写 `data['d1']['score']`。

## 事实 3: RecommendationAgent 只扫 A 股

[agents/recommendation_agent.py](../agents/recommendation_agent.py) line 347-360 `load_universe`:

```python
def load_universe(self) -> List[str]:
    """A 股 active 全量。"""
    rows = self.db.query(
        """SELECT DISTINCT stock_code FROM related_stocks
           WHERE market = 'A' AND status = 'active'
             AND confidence != 'staging'
             AND stock_code IS NOT NULL
             AND length(stock_code) = 6"""
    )
```

**`WHERE market='A'` 硬编码**。

`related_stocks` 表里有 KR 11 + US 40 + HK 1 + JP 1 共 **53 个标的被拒之门外**，永远不会被 RecommendationAgent 评分。

## 综合含义

**Scout 用户投资范围（中韩美 3 市场）与 Scout 实际能力（A 股为主）严重错配**。

这是 [docs/Scout_设计意图与实现偏离.md](Scout_设计意图与实现偏离.md) 第八节"市场缺口识别"的具体证据之一：

- AI 链美股（NVDA / AVGO / ANET 等）即使在 `related_stocks` 里，**也因财务空缺无法评分**
- KR 半导体链（삼성전자 / SK하이닉스 / 한미반도체 等）同上
- 即便 TD-020 实装三市场差异权重，**没有财务数据也算不出新分数** → KR/US 权重纸面存在，实际无效

**真正阻塞次序**:

```
TD-019 (跨市场链路 schema/data)
  └─ Phase 2A KR/US 财务采集 (FinancialAgent 扩展 OpenDART + yfinance)
      └─ 重新跑 RecommendationAgent (扩 universe to A+KR+US)
          └─ TD-020 (三市场差异权重) 才能真正生效
              └─ TD-014b (d6 估值 veto) 配套实装
```

TD-020 不能单独修，必须前置 TD-019 + Phase 2A 财务采集。

## 蓝图阈值的语义陷阱（2026-04-25 TD-014a 暴露）

蓝图阈值（如 v1.52 "<6"）写的是 **v1.52 当时权重语境**（CN d4=15）。当前代码用相对权重（d4=10），同样的"<6"在两个权重下对应的 base score 阈值不同。

**任何把蓝图阈值直接照抄成代码常量的地方都要 audit** —— 可能藏着别的偏离（不仅是 TD-014a 这一条）。

**更深的语义陷阱**: 蓝图阈值假设理想信号覆盖。实战中 Scout 自身覆盖能力可能没达到蓝图假设，导致按蓝图字面实装等于把 **"Scout 自身能力不足"惩罚到推荐质量上**。

### 规则

实施任何蓝图规则前，**先 verify "蓝图前提是否在实战中成立"**：

- 不成立 → 调蓝图 / 加前提条件 / 推迟到前提具备后实装
- **不要直接照抄**

### 实证 (TD-014a)

- 蓝图 v1.52 d4<6 假设: d4 信号采集覆盖完整，d4=20 反映"行业产出弱"
- 实战实际: d4_base=20 反映"Scout 信号采集不足"（TD-003 paper_d4 每天 27 errors / V1/V3 月频）
- 实装结果: 24 条 B 级会被降为 candidate，但这 24 条不是因为行业弱，是因为 Scout 没采到信号
- 决策: 推迟到 v1.69 unknowns 表上线（Phase 4-5）后再实装，配合"区分'没采到' vs '真的弱'"的能力
